## Context

本项目为全新开发，目标是构建一个 Web 应用，帮助用户将阿里云 CLB 配置迁移到腾讯云 CLB。当前无任何现有代码基础。

核心约束：
- 源端（阿里云）全程只读，严禁调用写入 API
- 目标端（腾讯云）所有写操作必须用户二次确认
- 支持断点续传，迁移状态需持久化
- 分步向导式 UI，用户友好

## Goals / Non-Goals

**Goals:**
- 提供简洁、可视化的分步迁移 Web 界面
- 实现阿里云 CLB 配置的只读拉取和解析
- 实现配置字段到腾讯云 CLB 的自动映射和不兼容检测
- 实现腾讯云 CLB 的受控写入（二次确认）
- 支持实时进度、操作日志、断点续传
- 生成清晰的迁移报告

**Non-Goals:**
- 本阶段不实现 LB 实例的创建/映射（仅使用已有的腾讯云 CLB 实例）
- 本阶段不实现 SSL 证书迁移
- 本阶段不实现后端服务器（RS）迁移
- 不实现多用户/多租户体系
- 不实现移动端适配（仅桌面端和平板端）

## Decisions

### 决策 1：技术栈选择

**选择**：后端 Python 3.10+ + Flask，前端原生 JavaScript（ES6+）+ Materialize CSS，构建工具 Vite

**理由**：
- 与已有 lb 项目保持技术栈一致，团队无需额外学习成本
- Python + Flask：轻量级 Web 框架、Blueprint 模块化路由、阿里云和腾讯云均有完善的 Python SDK
- 原生 JavaScript（ES6 模块化）：无框架依赖、首屏加载快、浏览器原生支持
- Materialize CSS：Material Design 风格、组件丰富（Steps 导航可通过自定义实现、Modal 弹窗、Progress 进度条、Cards 等）
- Vite 5.x：现代化构建工具，支持 HMR 热更新、代码分割、Terser 压缩、Gzip/Brotli 压缩
- 前后端分离：前端 SPA 单页应用，后端提供 RESTful API
- 所有静态资源本地化部署，不依赖 CDN，确保内网环境可用


### 决策 2：前后端通信方式

**选择**：RESTful API + WebSocket（用于实时日志和进度推送）

**理由**：
- 常规操作（凭证验证、配置拉取、迁移计划生成）使用 REST API
- 迁移执行过程中的实时进度和日志推送使用 WebSocket，避免轮询
- WebSocket 断开后自动重连，配合服务端状态持久化实现断点续传
- Flask 通过 flask-sock 或 gevent-websocket 支持 WebSocket

**备选方案**：
- 纯 REST 轮询：实现简单但延迟高、资源浪费
- SSE（Server-Sent Events）：单向推送可行但 WebSocket 双向通信更灵活（可暂停/继续）

### 决策 3：迁移状态持久化

**选择**：MySQL 8.0，Docker 容器化部署

**理由**：
- MySQL 8.0：成熟的关系型数据库，支持事务、JSON 字段类型、外键约束
- 所有数据统一存储在 MySQL 中：迁移任务状态、配置快照、映射结果、迁移报告、操作日志
- 支持断点续传的原子操作（事务保证）
- Docker Compose 编排：MySQL 容器 + Flask 应用容器，一键启动
- 迁移历史全部保留，不自动清理，MySQL 轻松支持
- 使用 SQLAlchemy ORM + Flask-Migrate（Alembic）管理数据库迁移

**备选方案**：
- SQLite：单文件数据库，不适合容器化场景（数据卷挂载问题）
- PostgreSQL：功能更强但 MySQL 生态更广、运维成本更低
- JSON 文件：无事务支持，不适合状态持久化

### 决策 4：凭证安全方案

**选择**：凭证仅存在于服务端内存（会话级），不持久化到磁盘

**理由**：
- AK/SK 为高敏感信息，不应写入数据库或文件
- 使用 Flask 服务端会话（内存存储）保存凭证，会话过期后自动清除
- 前端通过 HTTPS 传输凭证到后端，后端日志中屏蔽凭证字段
- 使用 python-dotenv 管理环境变量，.env 文件加入 .gitignore

### 决策 6：配置映射引擎架构

**选择**：基于规则的映射引擎，字段级映射 + 枚举值转换表

**理由**：
- 每个配置项定义映射规则（源字段 → 目标字段 + 转换函数）
- 枚举值映射使用静态映射表（如调度算法：wrr→WRR, wlc→LEAST_CONN）
- 不可映射的字段自动标记为"不兼容"并记录原因
- 映射规则可扩展，后续增加新字段映射无需修改引擎核心

### 决策 7：迁移执行流程

**选择**：任务队列 + 逐项执行 + 状态机

**理由**：
- 迁移计划拆解为有序的操作项列表（任务队列）
- 逐项执行，每项执行前发 WebSocket 通知（等待用户确认）
- 每项执行后立即持久化状态（pending/running/success/failed/skipped）
- 断点续传时从第一个非 success 项继续

## Risks / Trade-offs

- **[风险] 阿里云和腾讯云 API 限频** → 缓解：实现请求限速和重试机制，批量操作间加延迟
- **[风险] 配置映射遗漏** → 缓解：首先覆盖核心字段（协议、端口、调度算法、健康检查、转发规则），后续迭代补充；不可映射项明确标记
- **[风险] WebSocket 连接不稳定** → 缓解：客户端自动重连 + 服务端状态持久化，重连后同步最新状态
- **[风险] 凭证 .env 文件泄露** → 缓解：.env 加入 .gitignore 和 .dockerignore，文档提醒用户保护文件权限
- **[取舍] MySQL 需额外容器** → Docker Compose 统一编排，一键启动，运维成本可控
- **[取舍] 二次确认弹窗影响效率** → 提供批量确认模式平衡安全性和效率

## 补充决策（需求澄清后新增）

### 决策 8：腾讯云实例模式

**选择**：支持"使用已有实例"和"自动创建新实例"两种模式，用户在步骤中选择

**理由**：
- 使用已有实例：适合目标端已有基础设施的场景
- 自动创建实例：适合全新迁移场景，减少手动操作
- 创建实例属于写操作，同样需要二次确认弹窗

### 决策 9：迁移配置范围（高级参数）

**选择**：除基础配置外，还包括连接超时/空闲超时、带宽限制、访问控制等高级参数

**理由**：
- 连接超时/空闲超时：影响服务稳定性，必须迁移
- 带宽限制：影响流量控制，直接关系业务性能
- 访问控制（ACL/安全组策略）：影响安全策略，必须迁移
- 对于无法一一映射的高级参数，标记为"不兼容"并提醒用户

### 决策 10：地域范围

**选择**：仅支持中国大陆地域

**理由**：
- 当前迁移场景主要面向国内业务
- 阿里云仅拉取中国大陆地域的 CLB 实例列表
- 腾讯云仅展示中国大陆地域的可用区
- 后续可扩展支持港澳台和海外地域

### 决策 11：Docker 部署

**选择**：Docker Compose 编排（Flask 应用 + MySQL 8.0）

**理由**：
- 环境一致性：开发、测试、生产环境一致
- 简化部署：一键 docker-compose up 启动（Flask 应用容器 + MySQL 容器）
- 便于分发：打包为镜像，无需手动安装 Python/Node.js/MySQL 依赖
- Dockerfile 分阶段构建：前端 Vite build → 后端 Flask 运行
- MySQL 数据通过 Docker Volume 持久化，容器重启不丢失
- 支持 docker-compose.dev.yml 开发模式（热加载 + MySQL 暴露端口）

### 决策 12：迁移历史保留策略

**选择**：全部保留，不自动清理

**理由**：
- 迁移记录数据量小，不会造成存储压力
- 保留完整历史便于审计和回溯
- 用户可手动删除不需要的历史记录

### 决策 13：实例映射关系

**选择**：仅支持一对一和多对一映射，一对多暂不支持

**理由**：
- 一对一：最常见场景，实现简单
- 多对一：合并场景，需要端口冲突检测
- 一对多：拆分依据复杂（按监听器/域名/端口），暂不实现

### 决策 14：自动创建实例参数

**选择**：用户填写完整参数（名称、网络类型、VPC、子网、项目等）

**理由**：
- 腾讯云 CLB 实例创建涉及 VPC/子网选择，不同于阿里云的网络结构，无法自动映射
- 用户必须手动指定目标网络环境，确保实例创建在正确的 VPC 和子网中
- 前端提供完整的实例创建表单，包含：实例名称、网络类型（公网/内网）、VPC 选择、子网选择、所属项目
- 创建前弹出二次确认弹窗，展示完整参数

### 决策 15：多对一端口冲突处理

**选择**：冲突时展示列表，用户逐项选择处理方式（覆盖/跳过/重命名端口）

**理由**：
- 多对一合并场景下，不同源端实例可能有相同端口的监听器
- 直接阻止合并体验差，静默跳过不安全
- 展示冲突列表让用户逐项决策最透明：
  - 覆盖：用新配置覆盖已有配置（需二次确认）
  - 跳过：保留目标端已有配置
  - 重命名端口：将监听端口改为用户指定的新端口
- 冲突处理结果记入迁移报告

### 决策 16：单项失败行为

**选择**：用户可在执行前选择失败处理模式："失败继续"或"失败暂停"

**理由**：
- 不同用户对风险容忍度不同
- "失败继续"模式：单项失败记录日志和报告，继续执行后续项，适合批量迁移
- "失败暂停"模式：单项失败后暂停执行，弹窗提示用户选择"重试/跳过/终止"，适合需要逐项确认的场景
- 默认推荐"失败暂停"模式（更安全）
- 在迁移执行步骤的"开始执行"按钮旁提供模式选择
